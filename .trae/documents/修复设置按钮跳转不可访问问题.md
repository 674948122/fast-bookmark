## 问题定位
- 点击处理在 [content.js](file:///Users/pandegong/my-project/fast-bookmark/content.js#L377-L415) 中使用 `window.open(chrome.runtime.getURL('options.html'), '_blank')`。
- 扩展已在 [manifest.json](file:///Users/pandegong/my-project/fast-bookmark/manifest.json#L33) 配置 `options_page: options.html`，文件存在：[options.html](file:///Users/pandegong/my-project/fast-bookmark/options.html)。
- 可能原因：弹窗拦截或兼容性问题、路径或命名不一致、事件未绑定（Shadow DOM 节点获取为 null）。

## 修改方案
1. 将设置页打开逻辑改为使用 `chrome.runtime.openOptionsPage()`，提升兼容性与稳定性。
2. 增加兜底：若 `openOptionsPage` 不可用或抛错，则回退到 `window.open(chrome.runtime.getURL('options.html'), '_blank')`。
3. 加入防空与绑定时机保护：在绑定前校验 `settingsBtn` 是否存在；确保 Shadow DOM 插入后再绑定。
4.（可选）对 Manifest V3 兼容：在 `manifest.json` 增加 `options_ui: { "page": "options.html", "open_in_tab": true }`，同时保留现有 `options_page` 以兼容旧环境。

## 具体改动点
- 编辑 [content.js](file:///Users/pandegong/my-project/fast-bookmark/content.js#L377-L415)：
  - 替换点击回调为：
    - 先调用 `chrome.runtime.openOptionsPage()`；
    - `catch` 或 `undefined` 时使用 `window.open(chrome.runtime.getURL('options.html'), '_blank')`。
  - 绑定前增加 `if (!settingsBtn) return;` 或延迟绑定至容器挂载完成。
- 编辑 [manifest.json](file:///Users/pandegong/my-project/fast-bookmark/manifest.json#L33)：
  - 可选新增 `options_ui` 字段以对齐 MV3 规范。

## 验证步骤
- 重新加载扩展，在目标页面点击设置按钮：应直接打开 options 页，不受弹窗拦截影响。
- 在 Chrome 与 Firefox（如有）各验证一次；确认两种路径均可访问。
- 控制台无错误日志；检查 `settingsBtn` 非空与事件已绑定。

## 回滚与风险
- 改动范围小、可控；如不满足预期，恢复为原有 `window.open(getURL)` 实现即可。
- 增加的 `options_ui` 为可选项，不影响既有行为。
